## Phase Plan (5 Phases, ~2-3 days each)

### Phase 1: Foundation Layer (Day 1-2)
*Quick wins + foundational schema changes*

| Feature | Effort | Description |
|---------|--------|-------------|
| **F10: Memory Export / Prompt Formatting** | S | `as_prompt()` method + MCP tool. Template system for different LLM formats (Claude XML, ChatML, raw text). Configurable context budget. |
| **F4: Multi-Level Memory Tiers** | M | Schema: add `tier` field (working/short/long). Tier-specific defaults: working (TTL 1h), short (TTL 7d), long (no expiry). Auto-tier assignment heuristics. Migration for existing memories → long-term. |
| **F5: Importance Scoring + Adaptive Decay** | M | Schema: add `importance_score` (float), `access_count`, `last_accessed_at`. Decay function (exponential, configurable half-life per tier). Score boost on access. Recall results weighted by time-adjusted score. Extends existing upvote/downvote into continuous scoring. |

**Schema changes in this phase:**
```sql
ALTER TABLE memories ADD COLUMN tier VARCHAR(10) DEFAULT 'long';
ALTER TABLE memories ADD COLUMN importance_score FLOAT DEFAULT 1.0;
ALTER TABLE memories ADD COLUMN access_count INT DEFAULT 0;
ALTER TABLE memories ADD COLUMN last_accessed_at TIMESTAMP;
```

**Why first:** These are prerequisite fields that F3 (consolidation) and F2 (fact extraction) will need. F10 is a standalone quick win that unblocks prompt formatting immediately.

---

### Phase 2: Intelligence Layer (Day 3-4)
*LLM-powered extraction and enrichment*

| Feature | Effort | Description |
|---------|--------|-------------|
| **F6: Metadata Enrichment** | M | LLM-powered auto-extraction on `remember()`: topics, sentiment, entities, categories. Configurable (on/off, model choice). Enriched metadata stored in existing `metadata` JSONB field. Batch enrichment endpoint for backfill. |
| **F9: Dialog Classification** | M | Classify incoming content by intent (question/statement/instruction/preference), domain (technical/personal/business), and emotion. Classification stored in metadata. Filterable on `recall()` (e.g., `recall("...", intent="preference")`). |
| **F2: Fact Extraction + Conflict Resolution** | L | LLM extracts atomic facts from content on `remember()`. Facts compared to existing memories: ADD (new fact) / UPDATE (supersedes old) / DELETE (contradicts) / NOOP. Contradiction log for audit. Configurable extraction model. Old facts marked `invalidated_by` with reference to new memory ID. |

**New concepts:**
- `enrichment_pipeline`: Pluggable chain of enrichment steps (classify → extract entities → extract facts → score importance)
- `conflict_log` table: Records fact invalidations for audit trail
- All LLM calls optional — works without LLM (just stores raw, like today)

**Why second:** These features add intelligence to ingestion. They need the schema from Phase 1 (importance scoring, tiers) but don't need the graph layer yet.

### Phase 3: Graph Layer (Day 5-7)
*The big differentiator*

| Feature | Effort | Description |
|---------|--------|-------------|
| **F1: Knowledge Graph Layer** | XL | Entity + relationship extraction from memories. Graph stored in Postgres (adjacency table or Apache AGE extension). Hybrid retrieval: vector similarity + graph traversal combined scoring. New MCP tools: `graph_query`, `related_memories`, `entity_map`. Visualization endpoint (D3-compatible JSON). |

**Architecture options (decide during PM phase):**
1. **Pure Postgres:** Adjacency tables (`entities`, `relationships`) + recursive CTEs for traversal
2. **Apache AGE:** Postgres extension adding Cypher query support (graph-native, no separate DB)
3. **Separate graph DB:** Neo4j/Memgraph alongside Postgres (more complex infra)

**Recommendation:** Apache AGE or pure Postgres adjacency tables. Keeps single-DB simplicity (key differentiator vs Mem0/Memary which require Neo4j).

**Schema additions:**
```sql
CREATE TABLE entities (
    id UUID PRIMARY KEY,
    name TEXT NOT NULL,
    entity_type TEXT,  -- person, tool, concept, project, etc.
    metadata JSONB,
    first_seen_at TIMESTAMP,
    last_seen_at TIMESTAMP,
    mention_count INT DEFAULT 1
);

CREATE TABLE relationships (
    id UUID PRIMARY KEY,
    source_entity_id UUID REFERENCES entities(id),
    target_entity_id UUID REFERENCES entities(id),
    relation_type TEXT,  -- "uses", "depends_on", "prefers", "works_with"
    weight FLOAT DEFAULT 1.0,
    valid_from TIMESTAMP,
    valid_until TIMESTAMP,  -- temporal edges (Zep-style)
    memory_id UUID REFERENCES memories(id),  -- source memory
    metadata JSONB
);
```

**Graph-enhanced recall:**
```python
# Recall with graph context
results = lore.recall("deployment issues", graph_depth=2)
# Returns: matching memories + related entities + connected memories up to 2 hops
```

**Why third:** Most complex feature. Benefits from enrichment pipeline (Phase 2) feeding entities into the graph automatically.

### Phase 4: Operations Layer (Day 8-9)
*Ingestion, consolidation, maintenance*

| Feature | Effort | Description |
|---------|--------|-------------|
| **F7: Webhook / Multi-Source Ingestion** | M | REST endpoint: `POST /ingest` with auto-embedding + enrichment pipeline. Source adapters: Slack webhook, Telegram bot forward, Git commit hook, raw text POST. Source tracking (which adapter, channel, timestamp). Rate limiting + queue for burst ingestion. |
| **F3: Memory Consolidation / Auto-Summarization** | L | Background job: periodically scan memories by tier/age. Consolidation strategies: merge similar, summarize old clusters, compress episodic→semantic. Configurable schedule (daily/weekly). Consolidation log (what was merged, originals preserved or archived). Deduplication: detect near-duplicate memories (cosine similarity > 0.95), merge or flag. |

**Consolidation pipeline:**
```
1. Identify candidates (old, low-importance, similar clusters)
2. Group by topic/entity (using graph relationships)
3. LLM summarizes each group into condensed memory
4. Archive originals (soft delete with reference to consolidated memory)
5. Update graph edges to point to consolidated memory
6. Log everything for audit
```

**Why fourth:** Consolidation benefits hugely from the graph (Phase 3) for intelligent grouping. Webhook ingestion is independent but pairs naturally with the enrichment pipeline (Phase 2).

## MCP Tool Surface (v0.6.0)

### Existing (keep)
- `remember` — store memory (now with enrichment pipeline)
- `recall` — semantic search (now with graph-enhanced retrieval)
- `forget` — delete memory
- `list` — list memories (now filterable by tier, classification, entity)
- `stats` — memory statistics (now includes graph stats)
- `upvote` / `downvote` — relevance feedback (now feeds importance score)

### New Tools
- `as_prompt` — export memories formatted for LLM context injection
- `graph_query` — traverse knowledge graph (find related entities/memories)
- `related` — find memories connected via graph relationships
- `consolidate` — trigger manual consolidation of a memory cluster
- `enrich` — manually trigger enrichment on existing memories
- `ingest` — webhook-style ingestion with source tracking
- `classify` — classify a piece of text (returns intent/domain/emotion)
- `entity_map` — get entity graph for a topic/project

### Total: 14 MCP tools (up from 7)

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| LLM costs for enrichment | All LLM features optional + configurable model (use cheap models for classification) |
| Graph complexity | Stay Postgres-only (no Neo4j dependency). Start with adjacency tables, upgrade to AGE if needed |
| Scope creep | Strict BMAD per phase. Ship each phase independently. |
| Performance regression | Enrichment is async/optional. Graph queries have configurable depth limits. |
| Breaking changes | No users. But still version bump to 0.6.0 with migration script. |

## Success Criteria

After v0.6.0, Lore should be able to:
1. ✅ Store a memory and automatically extract entities, facts, topics, sentiment
2. ✅ Detect when a new fact contradicts an old one and resolve it
3. ✅ Answer "what's related to X?" using graph traversal, not just vector similarity
4. ✅ Auto-consolidate old memories into concise summaries
5. ✅ Ingest from Slack/Telegram/Git webhooks with zero config
6. ✅ Export memories as optimized LLM prompts
7. ✅ Work with any MCP-compatible AI tool out of the box
8. ✅ All of the above with a single `docker compose up`
