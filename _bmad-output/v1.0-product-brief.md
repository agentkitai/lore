# Lore v1.0 — "Total Recall" Epic Plan

**Version:** v1.0.0
**Codename:** Total Recall
**Date:** March 14, 2026
**Goal:** Transform Lore from a memory backend into a visible, trustworthy, cross-platform AI brain.

**Inspired by:** Obsidian + Claude Code "Second Brain" prompt (dontsleeponai.com) — validated the problem space, showed what users want. We have stronger infrastructure; now we need the UX and visibility layer.

---

## Platform Support Matrix

All 6 epics must work across all supported platforms:

| Platform | Memory Write | Memory Read | Integration Point |
|----------|-------------|-------------|-------------------|
| **OpenClaw** | lore-ingest hook | lore-retrieve hook (auto-inject) | Hooks + MCP |
| **Claude Code** | Claude Code ingest daemon | MCP server (manual search) | CLAUDE.md + MCP |
| **Codex** | MCP tools | MCP tools | MCP |
| **Cursor** | MCP tools | MCP tools | .cursorrules + MCP |

**Key insight:** OpenClaw has the strongest integration (auto-inject via hooks — agent never has to "decide" to search). Claude Code/Codex/Cursor rely on MCP (agent must choose to call the tool). Epics should strengthen both paths.

## E1: Graph Visualization (Web UI)

**Priority:** 1 (highest)
**Impact:** Marketing + user understanding
**Effort:** Medium-Large

### Why
Obsidian's Graph View is their single best marketing asset. One screenshot sells the product. Lore has knowledge graph data (v0.6.0 F1) but zero visual layer. Users can't see what's in their brain.

### What
A lightweight web UI that visualizes:
- Memory nodes as a force-directed graph
- Knowledge graph connections (entities, relationships)
- Clusters by project/topic/time
- Click-to-expand: see the actual memory content
- Filter by: project, type, date range, importance tier
- Search from the UI (semantic + keyword)

### Technical Scope
- New `lore ui` CLI command → starts local web server
- Frontend: Single-page app (vanilla JS or lightweight framework — no heavy deps)
- Backend: New REST endpoints on the existing Lore server
  - `GET /v1/graph` — nodes + edges for visualization
  - `GET /v1/graph/clusters` — grouped by topic/project
  - `GET /v1/memories/:id/connections` — single memory's connections
- D3.js or Sigma.js for graph rendering
- Responsive — works on desktop and mobile
- Optional: embed in Obsidian via iframe (for users who want both)

### Acceptance Criteria
- [ ] `lore ui` opens browser with graph visualization
- [ ] Memories appear as nodes, knowledge graph edges as connections
- [ ] Click a node → see full memory content + metadata
- [ ] Filter by project, type, date range
- [ ] Search bar with semantic search
- [ ] Works with 1000+ memories without performance issues
- [ ] Screenshot-ready for marketing (looks good)

## E2: Recent Activity Summary (Session Context)

**Priority:** 2
**Impact:** Continuity across sessions
**Effort:** Small

### Why
Auto-inject retrieves *relevant* memories based on the current query. But it misses *recent* context that doesn't semantically match. If you discussed architecture yesterday and ask about testing today, the architecture decisions won't surface. This is the "cold start with wrong context" problem.

### What
A "what happened recently" summary automatically available at session start:
- Last 24h of memories, regardless of semantic relevance
- Grouped by project/topic
- Concise — key decisions, changes, open items
- Available as both auto-inject (OpenClaw) and MCP tool (all platforms)

### Technical Scope
- New MCP tool: `lore.recent_activity` — returns last N hours of memories, summarized
  - Parameters: `hours` (default 24), `project` (optional), `format` (brief/detailed)
- New hook enhancement for OpenClaw: `lore-retrieve` also includes recent activity block
- For Claude Code/Cursor: Add to CLAUDE.md/.cursorrules protocol: "Call lore.recent_activity at session start"
- Summary generation: Use existing memories, group by project, extract key points
  - LLM-optional: structured grouping works without LLM, summary can use LLM if available

### Acceptance Criteria
- [ ] `lore.recent_activity` MCP tool returns last 24h memories grouped by project
- [ ] OpenClaw hook auto-includes recent activity in context injection
- [ ] Works without LLM (structured output) — enhanced with LLM if available
- [ ] Claude Code / Cursor setup docs updated
- [ ] Respects project scoping (only show relevant project's recent activity)

## E3: Pre-Compaction Hook (Context Rescue)

**Priority:** 3
**Impact:** Prevents memory loss during long sessions
**Effort:** Small-Medium

### Why
When a conversation gets long, context windows compress/compact. Decisions made early in the session get lost. The Obsidian prompt has a "PreCompact" hook that reminds Claude to save important stuff. We can do better — actually save it automatically.

### What
An OpenClaw hook that fires before compaction and auto-saves the current conversation's key information to Lore. For non-OpenClaw platforms, provide a protocol instruction.

### Technical Scope
- **OpenClaw:** New hook `lore-precompact` on `session:compacting` event (or equivalent)
  - Extracts: key decisions, action items, current task state, important context
  - Saves as a special memory type: `type: "session_snapshot"`
  - Tags with session ID for traceability
  - On next session start, these snapshots surface via auto-inject
- **Claude Code:** Add to CLAUDE.md: "When you notice context getting long, call `lore.save_snapshot`"
- **Codex/Cursor:** Same MCP tool available
- New MCP tool: `lore.save_snapshot`
  - Parameters: `content` (what to save), `session_id` (auto-detected if possible)
  - Stores with high importance tier so it surfaces reliably

### Acceptance Criteria
- [ ] OpenClaw hook fires before compaction, saves conversation snapshot to Lore
- [ ] `lore.save_snapshot` MCP tool available for all platforms
- [ ] Snapshots surface in next session's auto-inject
- [ ] Session snapshots have special type for easy filtering
- [ ] Works without LLM (saves raw content) — enhanced with LLM (extracts key points)

## E4: Topic Notes / Auto-Summaries (Concept Hubs)

**Priority:** 4
**Impact:** Knowledge organization + discovery
**Effort:** Medium

### Why
As memories accumulate, recurring concepts emerge: projects, people, tools, decisions. The Obsidian prompt auto-creates "topic notes" — hub pages that link related content. Lore's knowledge graph (F1) stores entity relationships, but there's no user-facing "here's everything about Project X" view.

### What
Auto-generated topic summaries that aggregate everything Lore knows about a recurring concept:
- Surfaces when a concept appears across 3+ memories
- Includes: first mention, key decisions, related entities, timeline
- Available via CLI, MCP, and web UI (E1)
- Can be periodically regenerated as new memories arrive

### Technical Scope
- New command: `lore topics` — list auto-detected topics with memory counts
- New command: `lore topics <name>` — show full topic summary
- New MCP tool: `lore.topics` — list topics, `lore.topic_detail` — get one topic
- Backend:
  - Query knowledge graph for entities with 3+ memory connections
  - Aggregate memories mentioning each entity
  - Generate summary (LLM-optional: without LLM, list memories chronologically; with LLM, produce narrative summary)
  - Cache summaries, regenerate on new ingestion
- Web UI integration: Topics as a sidebar/nav element in E1's graph view
- New API endpoints:
  - `GET /v1/topics` — list topics with counts
  - `GET /v1/topics/:name` — topic detail with linked memories

### Acceptance Criteria
- [ ] `lore topics` lists auto-detected topics (entities appearing in 3+ memories)
- [ ] `lore topics <name>` shows aggregated summary
- [ ] MCP tools available for all platforms
- [ ] Topics visible in web UI (E1) as navigation
- [ ] Threshold configurable (default 3)
- [ ] Works without LLM — enhanced with LLM for narrative summaries

## E5: Export / Snapshot (Safety Net)

**Priority:** 5
**Impact:** Trust + migration + backup
**Effort:** Small

### Why
The Obsidian prompt commits to git before any modification. Users trust systems they can back up and roll back. Lore stores everything in Postgres — users need confidence they can export, back up, and migrate.

### What
CLI commands for full export/import of Lore data:
- Export all memories to portable format (JSON, Markdown, or both)
- Import from export file (restore/migrate)
- Snapshot command for quick backup before risky operations
- Markdown export for use in Obsidian or any note-taking tool (bridge to their ecosystem)

### Technical Scope
- `lore export` — dump all memories + knowledge graph to JSON file
  - Options: `--format json|markdown|both`, `--project`, `--since`, `--output`
  - Markdown format: one file per memory, frontmatter metadata, folder structure by project/type
- `lore import` — load from export file
  - Deduplicates by content hash (safe to re-import)
  - Options: `--file`, `--project`, `--dry-run`
- `lore snapshot` — quick backup to `~/.lore/snapshots/YYYY-MM-DD-HHMMSS/`
  - Just wraps `lore export --output <snapshot-dir>`
- `lore restore` — restore from snapshot
  - Options: `--list` (show available snapshots), `--snapshot <name>`
- MCP tools: `lore.export`, `lore.snapshot` (so agents can trigger backups)

### Acceptance Criteria
- [ ] `lore export` produces complete JSON export
- [ ] `lore export --format markdown` produces Obsidian-compatible vault structure
- [ ] `lore import` restores from export, deduplicates safely
- [ ] `lore snapshot` / `lore restore` work for quick backup/rollback
- [ ] Export includes knowledge graph edges, not just memories
- [ ] Round-trip: export → wipe → import = identical data

## E6: Approval UX for Discovered Connections (Trust Layer)

**Priority:** 6
**Impact:** Data quality + user trust
**Effort:** Small-Medium

### Why
The Obsidian prompt asks users to approve topic notes before creating them. This builds trust — the system shows you what it found, you confirm. Lore's knowledge graph auto-creates connections, but users never see or validate them. Bad connections degrade retrieval quality silently.

### What
A review/approval flow for auto-discovered knowledge graph connections:
- Periodically surface new connections for review
- User confirms (keeps), rejects (removes), or edits (corrects)
- Rejected patterns are learned (don't re-suggest)
- Available via CLI, web UI, and as a periodic digest

### Technical Scope
- New state for knowledge graph edges: `status: pending|approved|rejected`
  - Default behavior (backward compat): all edges auto-approved (current behavior)
  - Opt-in: `lore config set graph.approval required` → new edges start as `pending`
- `lore review` CLI — interactive review of pending connections
  - Shows: entity A ←relationship→ entity B (from memory X)
  - Actions: approve (y), reject (n), skip (s), edit (e)
- Web UI integration: Review queue in E1's UI
- Digest feature: `lore.review_digest` MCP tool
  - Returns pending connections grouped by topic
  - Agent can present them conversationally: "I found these connections, want to keep them?"
- Rejected patterns stored to avoid re-suggesting same connection

### Acceptance Criteria
- [ ] Knowledge graph edges support `pending/approved/rejected` status
- [ ] `lore review` CLI provides interactive review
- [ ] Opt-in approval mode via config
- [ ] Rejected patterns not re-suggested
- [ ] Web UI shows review queue
- [ ] MCP tool enables agent-mediated review (conversational approval)
- [ ] Backward compatible: default mode auto-approves (no workflow change for existing users)
